# Project Rules

1. **Adding a program**: append one entry to the `$Lineup` array in `Setup-DevEnv.ps1` with `Key`, `Name`, `Group`, `Installed` (a `Test-AnyInstalled`-based detection scriptblock), and `Action` (the install call); add `Requires=@('choco')` if it needs Chocolatey. Verify the package ID exists before adding, and keep the file UTF-8 **with BOM**.
   - `Name`: a clean product name with **no parenthetical descriptions** вҖ” the category header already supplies the context (`'Bun'`, `'DBeaver'`, not `'Bun (JS runtime)'`); keep a parenthetical only for a meaningful variant (`'Node.js (LTS)'`). Use a plain string when it reads the same in both languages (almost always the case); use `@{ ko='...'; en='...' }` only for genuinely translatable wording such as config-action labels (e.g. `@{ ko='SSH нӮӨ мғқм„ұ'; en='SSH key generation' }`). Both forms are resolved by `Loc`.
   - `Group`: a **key** from the `$GroupNames` table (e.g. `'lang'`, `'editor'`, `'cloud'`), **not** a display string. To add a new category, add one entry to `$GroupNames` with both `ko` and `en` names.
   - The `-Name` passed to `Install-*` (shown in logs/summary) should be a language-neutral product name вҖ” normally identical to the display `Name`.
   - To make an item **opt-in** (unchecked by default even when *not* installed вҖ” e.g. EOL/niche tools like Python 2.7), add `DefaultOff=$true` to its `$Lineup` entry. The picker is a responsive auto-balancing grid, so adding items requires no layout changes вҖ” only the `$Lineup` entry (and a `$GroupNames` entry for a brand-new category).
2. **i18n (Korean / English)**: the script is bilingual, selectable via a top-right dropdown in the picker window and the `-Lang ko|en` switch (auto-detected from OS locale otherwise). Every user-facing message is keyed in the `$T` table and emitted through `L 'key'` вҖ” **never** hardcode a runtime string. When adding any new message, add it to **both** `$T.ko` and `$T.en` (keep them symmetric); use `{0}`/`{1}` placeholders with `(L 'key') -f $a, $b`. The only intentional Korean string literal outside the data tables is the `'н•ңкөӯм–ҙ'` dropdown label.
3. **Detection**: every item's `Installed` check must detect the tool no matter how it was installed (winget / choco / scoop / npm / official installer / MSI / portable / Store) вҖ” combine command + path + ARP + Appx signals via `Test-AnyInstalled`; never rely on `winget list` alone. Don't trust a single signal: ARP is **not guaranteed** (e.g. SQLyog registers no uninstall entry, so detect it by its exe path вҖ” `'...\SQLyog*\SQLyog*.exe'`), and a tool that can live in several places (e.g. the Android SDK under our `C:\Android` *or* Android Studio's `%LOCALAPPDATA%\Android\Sdk`) must be checked across **all** known roots, not just our own install dir.
4. **Install method**: before adding, evaluate whether the official recommended command (npm, official install script, etc.) or winget is the better fit for that tool, and choose accordingly rather than defaulting blindly.

> Always write `CLAUDE.md` in English, but reply to the user in the language the user is using.
